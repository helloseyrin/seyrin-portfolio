# Session — 2026-03-14

## Insights

### I001 — Separate API endpoints for different mutation types
The existing `/api/save` route uses path-based `setByPath` mutations — good for editing a single field but structurally unable to splice arrays. Adding/deleting items needs a full-replace endpoint instead. Keeping the two concerns separate (field patch vs. array replace) keeps each endpoint simple and purpose-specific rather than building a sprawling generic mutation API.

### I002 — Discriminated unions for multi-source tooltip data
Using `kind: "cert" | "experience" | "project"` as a discriminated union for `SkillSource` gives TypeScript full narrowing per branch — each section of the tooltip knows exactly what fields exist without optional property checks. Safer and more readable than a flat object with optional fields that may or may not be present depending on context.

### I003 — Data URI as icon fallback when SimpleIcons has no entry
When a brand doesn't exist in SimpleIcons, an inline SVG as a `data:` URI is the most reliable fallback — no external dependency, no silent failure, no need for error handling. Works as a regular `<img src>` value.

### I004 — Hook capture limitation: file changes yes, response text no
Claude Code hooks (PostToolUse, Stop) fire on tool events and receive JSON payloads via stdin. They can log which files were changed. They cannot capture Claude's response text — insight blocks written in chat are invisible to hooks. The workaround: write insights explicitly to the logbook file via tool call during the session, not just in chat output.

### I005 — Logbook as a two-layer system: automatic + intentional
The hook captures the *what* (which files changed, when) automatically. The *why* and *so what* — insights, decisions, trade-offs — require deliberate writes during the session. This mirrors good commit discipline: the diff shows what changed, the message explains the reasoning. A logbook that only has file timestamps is a changelog. One that also has reasoning is institutional memory.

### I006 — next-mdx-remote over @next/mdx for file-system content
When MDX files live outside the `app/` directory (e.g. `content/entries/`), `next-mdx-remote` is the right choice: it reads files at build/request time from an arbitrary path and supports frontmatter via `gray-matter`. `@next/mdx` in contrast bakes MDX directly into the Next.js page graph — great for in-app docs but unable to dynamically source from a `content/` folder. Using `next-mdx-remote/rsc` with React Server Components means zero client bundle cost: MDX renders on the server, no hydration needed for static prose.

### I007 — Next.js 15+ async params in dynamic routes
In Next.js 15, route `params` in `page.tsx` are now typed as `Promise<{...}>` and must be awaited before access. This is a silent breaking change from Next.js 14 — the old `{ params }: { params: { slug: string } }` pattern silently passes a Promise object at runtime. Always await params: `const { slug } = await params`.

### I008 — CSS custom property chain breaks when var() references a variable set on a child scope
`--font-mono: var(--font-fira-code), "Fira Code", monospace` defined in `:root` silently fails when `--font-fira-code` is only set on `body` (via a Next.js font class). Although CSS custom properties *should* resolve lazily at use-time, the `:root`-defined intermediate variable does not reliably pick up values injected on `body` in Turbopack's dev CSS pipeline. Fix: reference the Next.js font variable directly at every use site (`font-family: var(--font-fira-code), "Fira Code", monospace`) rather than through an intermediate token. Eliminates the silent inheritance failure.

### I009 — Font personality hierarchy: Outfit body, Fira Code for all mono/code contexts
The site now uses exactly two fonts: Outfit (300–600, sans-serif, all prose) and Fira Code (400–500, monospace, tags / code / meta rows / entry prose body / notepad titlebar). Three experimental note-body fonts were tried (Courier Prime → IBM Plex Mono → Space Mono) before the decision that Fira Code should be the single mono voice. Fewer font choices = more coherent aesthetic; a second mono font adds noise without adding meaning here.

### I010 — coverWord vs tags[0] for cover image watermark
Entry cards display a watermark word on the gradient cover placeholder. Initially this used `tags[0]`, which polluted the tag list with display-only words ("autodidactism" as a tag vs. "self-teaching" as an actual descriptor). Solution: dedicated `coverWord` frontmatter field — purely presentational, never shown in tag lists. Pattern generalises: display metadata (cover words, hero labels) should never be conflated with semantic metadata (tags, categories).
